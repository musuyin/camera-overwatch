## 一、项目背景与目标

### 背景

本项目为视觉感知课程大作业，结合计算机视觉核心技术，实现摄像头捕捉手势并映射为键鼠输入，操控《守望先锋》中的英雄进行游戏行为。

### 核心设计理念

**动作匹配感优先**：用户在摄像头前的手势动作，应尽量与游戏中英雄的动作在直觉上对应。英雄选型以"左右手各司其职、技能动作有直觉映射"为核心标准，不选择枪械型英雄（动作无区分度）。

### 不在范围内

- 不要求实战可用（无超低延迟要求）
- 不控制移动（WASD），专注于技能和攻击的手势映射
- 不识别手指精细姿态，仅用手部整体状态

## 二、可行性评估

### 技术可行性：✅ 高

| 环节 | 方案 | 风险 |
|------|------|------|
| 手部检测与追踪 | MediaPipe Hands（Google，实时，无需训练） | 低，开箱即用 |
| 手势识别 | 基于关键点几何规则（位置区域、伸缩、双手关系） | 低，规则清晰 |
| 键鼠模拟输出 | `pynput` | 低，成熟库 |
| 游戏验证 | 训练模式/自定义对局观察效果 | 中，需调试映射 |

### 可用识别维度

MediaPipe 输出每只手21个关键点，以下维度经过评估可稳定使用：

| 维度 | 提取方式 | 可靠性 |
|------|----------|--------|
| **左/右手区分** | MediaPipe handedness 自动标注 | ★★★★★ |
| **手的位置区域** | 手腕坐标落在画面哪个区域（上/中/下） | ★★★★★ |
| **手的伸缩（前推/收回）** | 手掌在画面中的面积大小，或手腕z轴深度 | ★★★★ |
| **移动方向** | 帧间手腕坐标位移向量 | ★★★★ |
| **手势形状** | 握拳 vs 张开（五指关键点收拢程度） | ★★★★ |
| **双手相对关系** | 双手距离（靠拢/分开）、双手是否交叉 | ★★★★ |
| **移动速度** | 帧间位移大小 | ★★★ 连续值，阈值敏感，避免作为主要维度 |

> **设计原则**：优先使用离散状态维度（伸/收、区域、交叉），避免以速度作为主要判断依据。

### 时间可行性：✅ 合理

预估总工时约 **25-30 小时**，可在 3 周内完成。

### 主要风险

| 风险 | 说明 | 应对 |
|------|------|------|
| 误触发 | 手抖或过渡姿态触发错误指令 | 加入状态持续确认（约100-150ms去抖窗口） |
| z轴深度不稳定 | 摄像头深度估计有误差 | 优先用手掌面积替代z轴判断伸缩 |
| 双手交叉检测 | 手腕坐标交叉判断需额外逻辑 | 用左右手腕x坐标大小关系判断是否交叉 |
| 游戏反外挂 | 自动输入可能被检测 | 仅在训练模式/自定义对局演示 |

## 三、英雄选型

选取 **2 个英雄**作为初始实现目标，覆盖不同识别维度，后续视工作量决定是否增加第三个。

---

### 英雄一：莫伊拉（Moira）—— 双手独立状态型

**演示重点**：左右手分别独立控制两种能量光束，天然对应 MediaPipe 双手独立识别。

**核心识别维度**：手的伸缩状态、双手同时状态

| 用户动作 | 识别维度 | 游戏指令 | 匹配感 |
|----------|----------|----------|--------|
| 左手向前伸出 | 左手伸缩：伸 | 持续治疗光束（左键按住） | ★★★★★ |
| 左手收回 | 左手伸缩：收 | 停止治疗（左键释放） | ★★★★★ |
| 右手向前伸出 | 右手伸缩：伸 | 持续伤害光束（右键按住） | ★★★★★ |
| 右手收回 | 右手伸缩：收 | 停止伤害（右键释放） | ★★★★★ |
| 任一手向侧面甩出 | 移动方向：横向 | 扔治疗球/伤害球（E） | ★★★★ |
| 双手同时向前伸出 | 双手同时伸出 | 大招聚合生命（Q） | ★★★★★ |

**技术亮点**：左右手状态完全独立维护，互不干扰，可同时触发（左手治疗 + 右手伤害）。

---

### 英雄二：拉玛莎（Ramattra）—— 位置区域 + 形态切换状态机型

**演示重点**：右手位置区域控制不同技能，双手交叉触发形态切换，两种形态下手势映射不同。

**核心识别维度**：手的位置区域（上/中/下）、双手交叉、状态机

#### 普通形态（Omnic Form）

| 用户动作 | 识别维度 | 游戏指令 | 匹配感 |
|----------|----------|----------|--------|
| 右手处于中部区域 + 向前伸 | 右手伸缩：伸，区域：中 | 普攻（左键） | ★★★★ |
| 右手抬至上部区域 | 右手位置区域：上 | 立盾牌（Shift） | ★★★★★ |
| 右手压至下部区域 | 右手位置区域：下 | 丢漩涡（E） | ★★★★★ |
| 双手手腕交叉 | 双手交叉 | 变身天罚形态（Q触发） | ★★★★★ |

#### 天罚形态（Nemesis Form）

| 用户动作 | 识别维度 | 游戏指令 | 匹配感 |
|----------|----------|----------|--------|
| 右手向前伸出 | 右手伸缩：伸 | 右拳攻击（左键） | ★★★★★ |
| 左手向前伸出 | 左手伸缩：伸 | 左拳攻击（右键） | ★★★★★ |
| 双手同时向前伸出 | 双手同时伸出 | 格挡（Shift） | ★★★★★ |
| 双手再次交叉 | 双手交叉 | 退出天罚形态（形态结束） | ★★★★ |

**技术亮点**：引入状态机（普通形态 / 天罚形态），同一手势在不同状态下映射不同指令，代码结构清晰，报告论述有层次。

## 四、技术栈

```
Python 3.10+
├── 视觉采集        OpenCV (cv2)          摄像头读帧、画面显示
├── 手部追踪        MediaPipe Hands       21点手部关键点实时检测、左右手标注
├── 动作识别        自定义几何规则模块      位置区域 / 伸缩 / 交叉 / 双手关系 规则引擎
├── 状态机          自定义 HeroState 模块  管理英雄形态切换（拉玛莎）
├── 键鼠模拟        pynput                键盘/鼠标事件注入
└── 调试可视化      OpenCV + numpy        实时绘制关键点、当前状态标签、触发指令
```

### 为什么不用深度学习分类器

手势类别少（约6-10个），且以位置/伸缩等离散状态为主要特征，规则引擎比训练分类器：
- 开发更快，无需标注数据集
- 可解释性强，便于报告说明技术细节
- 对演示环境泛化更稳定

## 五、系统架构

```
摄像头帧
  │
  ▼
OpenCV 读帧（BGR → RGB）
  │
  ▼
MediaPipe Hands
  ├── 左手：21个关键点坐标
  └── 右手：21个关键点坐标
  │
  ▼
手势识别模块（GestureRecognizer）
  ├── 位置区域判断   手腕y坐标 → 上/中/下
  ├── 伸缩判断       手掌面积或z轴 → 伸出/收回
  ├── 移动方向       帧间位移向量 → 方向
  ├── 双手交叉       左右手腕x坐标大小关系
  └── 去抖模块       状态持续N帧才确认触发
  │
  ▼
英雄映射模块（HeroMapper）
  ├── MoiraMapper    独立维护左手状态、右手状态
  └── RamattraMapper 状态机（Omnic/Nemesis） + 指令映射表
  │
  ▼
pynput 模拟输入 → 游戏接收
```

## 九、验收标准

1. **功能**：莫伊拉和拉玛莎各自可完整演示全部技能手势映射
2. **稳定性**：连续运行5分钟无崩溃，误触发率可接受（<30%）
3. **延迟**：手势到游戏响应 ≤ 300ms
4. **演示**：录制完整演示视频，展示动作匹配感
